# Internationalization (i18n) Guidelines

## Configuration

### Supported Languages

- **English (en)** - Default language, no URL prefix
- **French (fr)** - URL prefix `/fr/`

### URL Strategy

- `localePrefix: 'as-needed'` - Only non-default locales get prefixes
- English: `/waitlist`, `/dashboard`
- French: `/fr/waitlist`, `/fr/dashboard`
- API routes: `/api/*` (not localized)

### Locale Detection

Automatic detection in this order:

1. URL path (`/fr/waitlist`)
2. Cookie (`NEXT_LOCALE`)
3. Browser `Accept-Language` header
4. Default locale (`en`)

## File Structure

### Translation Files Location

```
messages/
├── en/                   # English translations
│   ├── common.json       # Shared (nav, footer, actions)
│   ├── waitlist.json     # Waitlist page
│   ├── admin.json        # Admin panel
│   └── [feature].json    # One file per feature/page
└── fr/                   # French translations
    ├── common.json
    ├── waitlist.json
    ├── admin.json
    └── [feature].json
```

### Naming Convention

- File names: `kebab-case.json` (e.g., `user-profile.json`)
- Match the feature/page name
- Keep files small and focused (one per page/feature)

## Translation Rules

### CRITICAL: Always Create Translations

**When creating a new page or component with user-facing text:**

1. ✅ **ALWAYS create translation files for BOTH languages**
   - Create `messages/en/[feature].json`
   - Create `messages/fr/[feature].json`

2. ✅ **ALWAYS use `useTranslations()` hook**

   ```typescript
   import { useTranslations } from 'next-intl';

   export default function MyPage() {
     const t = useTranslations('myFeature');
     return <h1>{t('title')}</h1>;
   }
   ```

3. ✅ **NEVER hardcode text in components**

   ```typescript
   // ❌ BAD - Hardcoded text
   <h1>Welcome to GateCtr</h1>

   // ✅ GOOD - Translated text
   const t = useTranslations('home');
   <h1>{t('welcome')}</h1>
   ```

4. ✅ **Update `i18n/request.ts` to load new translation files**
   ```typescript
   const messages = {
     common: (await import(`../messages/${locale}/common.json`)).default,
     waitlist: (await import(`../messages/${locale}/waitlist.json`)).default,
     myFeature: (await import(`../messages/${locale}/my-feature.json`)).default, // Add this
   };
   ```

### Translation File Structure

Use nested objects for logical grouping:

```json
{
  "page": {
    "title": "Page Title",
    "subtitle": "Page Subtitle"
  },
  "form": {
    "email": {
      "label": "Email",
      "placeholder": "you@company.com",
      "required": "Email is required",
      "invalid": "Invalid email"
    },
    "submit": "Submit",
    "submitting": "Submitting..."
  },
  "errors": {
    "network": "Network error",
    "failed": "Operation failed"
  }
}
```

### Usage Patterns

#### Basic Translation

```typescript
const t = useTranslations('waitlist.form');
<label>{t('email.label')}</label>
<input placeholder={t('email.placeholder')} />
```

#### With Parameters

```typescript
// messages/en/waitlist.json
{
  "success": {
    "message": "You're #{position} in line"
  }
}

// Component
const t = useTranslations('waitlist.success');
<p>{t('message', { position: 42 })}</p>
// Output: "You're #42 in line"
```

#### With Rich Text

```typescript
// messages/en/common.json
{
  "footer": {
    "copyright": "© {year} GateCtr. All rights reserved."
  }
}

// Component
const t = useTranslations('common.footer');
<p>{t('copyright', { year: new Date().getFullYear() })}</p>
```

#### Pluralization

```typescript
// messages/en/admin.json
{
  "users": {
    "count": "{count, plural, =0 {No users} =1 {1 user} other {# users}}"
  }
}

// Component
const t = useTranslations('admin.users');
<p>{t('count', { count: userCount })}</p>
```

## Component Guidelines

### Pages in `app/[locale]/`

All pages must:

1. Use `useTranslations()` hook
2. Have corresponding translation files
3. Include `LanguageSwitcher` component

```typescript
'use client';

import { useTranslations } from 'next-intl';
import { LanguageSwitcher } from '@/components/language-switcher';

export default function MyPage() {
  const t = useTranslations('myFeature');

  return (
    <div>
      <LanguageSwitcher />
      <h1>{t('page.title')}</h1>
      <p>{t('page.description')}</p>
    </div>
  );
}
```

### Server Components

```typescript
import { useTranslations } from 'next-intl';

export default function ServerPage() {
  const t = useTranslations('myFeature');

  return <h1>{t('title')}</h1>;
}
```

### Client Components

```typescript
'use client';

import { useTranslations } from 'next-intl';

export default function ClientComponent() {
  const t = useTranslations('myFeature');

  return <button>{t('action')}</button>;
}
```

## Navigation

### Using Localized Links

Always use the `Link` component from `i18n/routing`:

```typescript
import { Link } from '@/i18n/routing';

// Automatically adds locale prefix when needed
<Link href="/waitlist">Join Waitlist</Link>
// English: /waitlist
// French: /fr/waitlist
```

### Programmatic Navigation

```typescript
import { useRouter } from "@/i18n/routing";

const router = useRouter();
router.push("/dashboard"); // Automatically localized
```

### Getting Current Locale

```typescript
import { useLocale } from "next-intl";

const locale = useLocale(); // 'en' or 'fr'
```

## Form Validation

### Zod Schema with Translations

```typescript
import { useTranslations } from "next-intl";
import { z } from "zod";

export default function MyForm() {
  const t = useTranslations("myFeature.form");

  const schema = z.object({
    email: z.string().min(1, t("email.required")).email(t("email.invalid")),
    name: z.string().min(2, t("name.tooShort")),
  });

  // Use schema for validation
}
```

## Checklist for New Features

When creating a new page or feature:

- [ ] Create `messages/en/[feature].json` with all text
- [ ] Create `messages/fr/[feature].json` with French translations
- [ ] Update `i18n/request.ts` to import new translation files
- [ ] Use `useTranslations('[feature]')` in component
- [ ] Replace all hardcoded text with `t()` calls
- [ ] Add `LanguageSwitcher` to page header
- [ ] Test both `/path` (English) and `/fr/path` (French)
- [ ] Verify form validation messages are translated
- [ ] Check error messages are translated

## Best Practices

### DO ✅

- Create translation files immediately when creating pages
- Use nested keys for organization (`form.email.label`)
- Keep translation files small and focused
- Use parameters for dynamic content
- Test both languages before committing
- Use `common.json` for shared translations

### DON'T ❌

- Hardcode text in components
- Create monolithic translation files
- Mix languages in the same file
- Forget to update `i18n/request.ts`
- Skip French translations
- Use inline translations for repeated text

## Troubleshooting

### Translation Key Not Found

If you see the key path instead of translated text:

1. Check the translation file exists in both `en/` and `fr/`
2. Verify the key path matches exactly
3. Ensure `i18n/request.ts` imports the file
4. Restart dev server after adding new files

### Wrong Language Displayed

1. Check URL has correct prefix (`/fr/` for French)
2. Clear browser cookies (`NEXT_LOCALE`)
3. Check `i18n/routing.ts` configuration
4. Verify `proxy.ts` middleware is working

### Missing Translations

1. Always create both `en` and `fr` files
2. Keep structure identical in both files
3. Use TypeScript for type safety (optional)
